I know there have been other Rust talks on this conference and that's great. I'm presenting a new method I think that hasn't been shown before.

Motivation

  • Speed up your program
  • Escape the GIL (multithreading)
  • Bind to existing system-libraries
There are probably more reasons but I feel these are the main ones. Primarily the reason we don't write everything in Python is speed.

Ways to speed up Python

Manual/C: Python C extensions, ctypes, cffi

Binding generator: Boost.Python, SWIG, PyO3

Specialized lib: pillow, numpy

JIT: pypy

Python-like dialect: cython

You might disagree with my classifications but this is just how I think about it.

Problems of current approaches

Unsafe

C is not ergonomic

JIT unreliable

Multithreading

Why do we even need a new approach? It seems that we already have plenty of choices. Unsafe: Multithreading issues (dirty reads, dirty writes, what happens if you have multiple writers?) Unsafe: Segmentation faults because of invalid memory accesses. Unsafe: Initialized data C: Memory management Specialized libs only work for some problems. JIT is unpredictable without guarantees. Other dialects: It's not Python so you might as well do it properly.

Instead, oxidize your snake

What is Rust?

  • Safe
  • Modern
  • Fast
Safe: No null pointers, no multithreading issues Modern: Great unicode support, modern syntax, great tools Fast: C-like performance
